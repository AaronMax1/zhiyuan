# 高考报志愿项目整体优化方案

## 当前调研结论

本次参考了三个来源：

- `xuefeng-agent`：优势是聊天式交互、OpenAI 兼容模型配置、本地录取数据库、Tavily 联网搜索入口、面向普通用户的一键体验。弱点是推荐算法主要靠位次区间 SQL 分桶，概率、选科、家庭画像、风险解释不足。
- `gaokao-advisor`：优势是可计算推荐引擎，包含分数转位次、录取概率、冲稳保标签、效用评分、家庭画像、选科过滤、专业过滤、平行志愿模拟、数据审计等模块。弱点是前端体验和 Agent 解释层较弱，完整数据库不随仓库授权发布。
- `cn.colleges.chat` / `CollegesChat/university-information`：优势是学校生活质量信息丰富，包括宿舍、空调、门禁、洗浴、外卖、交通、校园网等招生简章不会写清的内容。许可是 CC BY-NC-SA 4.0，商业化使用要谨慎。

## 产品定位

目标不是做一个“会聊天的志愿填报玩具”，而是做一个“可解释、可审计、可追问”的志愿决策系统。

核心输出从单纯学校清单升级为：

- 录取概率：明确每个学校/专业的 `P(admit)`。
- 冲稳保结构：按风险配比生成候选。
- 专业路径判断：就业、考研、证书、家庭支撑周期、行业门槛。
- 学校生活质量：宿舍、校区、交通、作息、网络、门禁等。
- 数据来源标注：官方录取数据、学校生活质量库、联网搜索、模型推理分开标。
- 风险说明：数据缺口、年份波动、换制首年、计划变化、调剂风险单独提示。

## 推荐架构

建议采用“确定性推荐引擎 + LLM 解释/追问层”的结构。

### 1. 数据层

核心表建议拆成四类：

- `score_segments`：一分一段表，用于分数转位次。
- `admission_scores`：院校层历史录取最低分/最低位次。
- `major_scores`：专业层历史录取最低分/最低位次。
- `enrollment_plans`：当年招生计划，排除计划数为 0 的专业。

扩展表：

- `school_life_quality`：从 CollegesChat 结构化出宿舍、空调、卫浴、门禁、交通、外卖、校园网等字段。
- `school_aliases`：学校别名、校区名、历史更名，用于匹配 `北京邮电大学沙河校区` 这类条目。
- `source_registry`：记录每条数据来源、更新时间、许可证、可信度。

### 2. 推荐引擎

以 `gaokao-advisor` 为主干：

- 使用 `DataLoader.score_to_rank()` 处理分数转位次，并保留单调保护和缺失年份 fallback。
- 使用 `p_admit = 1 - Φ((student_rank - μ) / σ)` 计算录取概率。
- 使用阈值分桶：冲 `10%-45%`，稳 `45%-75%`，保 `75%+`。
- 使用 `utility_score` 把学校层次、城市偏好、专业偏好、家庭画像变成可调权重。
- 使用 `RecommendationEngine._select_slots()` 控制冲稳保比例和同校专业数量上限。
- 接入 `monte_carlo.simulate()` 输出未录取概率、落入层次概率、平均录取槽位。

`xuefeng-agent` 的原 `/recommend` 位次区间 SQL 可保留为“快速候选召回”或 fallback，但不应作为主推荐逻辑。

### 3. Agent 层

借鉴 `xuefeng-agent`：

- 保留聊天式入口、多轮上下文、API 设置、Tavily 搜索。
- LLM 不直接决定推荐结果，只负责：
  - 追问缺失信息：省份、科类/选科、分数/位次、专业偏好、城市偏好、家庭背景、风险偏好。
  - 调用推荐 API。
  - 把结构化结果解释成人话。
  - 标注来源和不确定性。

建议工具调用流程：

```text
用户输入
-> LLM 提取槽位
-> 缺信息则追问
-> /api/rank 分数转位次
-> /api/recommend 生成结构化推荐
-> 查询 school_life_quality
-> 可选 Tavily 搜最新招生计划/官网说明
-> LLM 汇总解释
```

## 学校信息接入方案

`cn.colleges.chat` 适合做“生活质量补充卡片”，不适合作为录取推荐主依据。

建议结构化字段：

- 宿舍：上床下桌、几人间、空调、独卫、浴室距离。
- 作息：早晚自习、晨跑、跑步打卡、门禁、查寝、断电断网。
- 生活：外卖、快递、食堂、超市、洗衣机、热水时间。
- 交通：校区位置、地铁、通勤距离。
- 信息可信度：是否归档、最后更新时间、贡献者说明、原文链接。

匹配策略：

- 先按标准学校名匹配。
- 再按校区名匹配。
- 再用别名表处理简称、旧名、学院更名。
- 匹配不到时只给外链搜索，不让 LLM 编造。

许可策略：

- 非商业内部项目：可缓存摘要，但需要署名、链接来源、保持相同许可要求。
- 商业项目：优先只做外链、用户按需查看；或联系授权；或自建问卷数据源。

## 前端体验

不要只做聊天窗口。建议一屏分三块：

- 左侧：用户画像表单，省份、科类、分数/位次、选科、家庭背景、风险偏好。
- 中间：推荐表格，冲稳保分组，展示概率、学校、专业、城市、层次、匹配原因。
- 右侧：解释面板，专业路径、学校生活质量、风险提示、数据来源。

关键交互：

- 分数输入后自动换算位次。
- 权重滑块：学校优先、专业优先、城市优先、就业优先。
- 风险模式：保守、稳健、进取。
- 每条推荐可展开：历年位次、概率曲线、专业解释、学校生活质量。
- 一键导出：志愿草表、家长版解释、风险清单。

## 实施阶段

### 第一阶段：可运行 MVP

- 用 `gaokao-advisor` 的 `engine/` 做核心推荐服务。
- 接上 `xuefeng-agent` 的聊天 UI 或重做一个轻量前端。
- 完成 `/api/rank`、`/api/recommend`、`/api/school-life` 三个接口。
- 学校生活质量先做离线缓存和学校名匹配。

### 第二阶段：解释和可信度

- 推荐结果每条加来源标签。
- 增加数据审计页：缺省份、缺年份、缺专业分、换制首年提示。
- LLM 输出固定结构：结论、冲稳保表、风险、下一步确认项。
- 加入 Monte Carlo 模拟。

### 第三阶段：数据增强

- 官方考试院数据 ETL。
- 招生计划更新。
- 学校生活质量定期同步。
- 学校别名/校区映射库。
- 就业、考研、专业资格路径数据。

### 第四阶段：产品化

- 用户方案保存。
- 多方案对比。
- 志愿顺序模拟。
- PDF/Excel 导出。
- 后台数据更新和质量审核。

## 优先级建议

最高优先级：

1. 以 `gaokao-advisor` 引擎替换简单 SQL 分桶。
2. 建立统一 SQLite schema 和数据来源登记。
3. 把 LLM 限定为解释层，不让它直接编推荐。
4. 接入学校生活质量卡片，但清晰标注来源和许可。

暂缓：

- 花哨人格化输出。
- 无来源联网总结。
- 商业化缓存 CollegesChat 全量内容。
- 只按裸分推荐。
